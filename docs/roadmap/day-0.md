# Day 0: 프롤로그

우연히 링크드인을 훑어보다가 짧은 포스트를 읽었습니다.

> "지금부터 라이브러리를 만들고, 프레임워크를 만드세요.  
> 10년 후에는 AI가 여러분을 대체할 겁니다."

요즘 개발을 하다 보면 문득 AI로 인한 위기감을 느낄 때가 있곤 합니다.

세상에 처음 ChatGPT가 출시되었을 때에 비해 이제는 Cursor든 Claude Code든 AI 도구의 사용과 성능을 공식적으로 인정해주는 분위기가 되어가고 있기 때문일까요.  
바이브코딩으로 비교적 손쉽게 서비스를 런칭하는 사례를 종종 접하게 되었기 때문일지도 모르겠습니다.  
스스로 이 방법이 최선일지, 왜 이렇게 동작하는지에 대한 근본적인 의문을 갖는 빈도가 점차 줄어들고 있다는 점도 한몫하고 있을 것입니다.

'위기감' 이라는 제 마음을 하나씩 풀어보자면,

어떤 추상화된 도구가 왜 그렇게 동작하고 구현되었는지보다는 **사용 패턴**에 집중하려 하고,  
AI의 추천보다 더 나은 설계나 대안이 있으리란 **고민의 빈도가 낮아졌으며**,  
의사결정의 근거나 배경지식의 출처 역시 점점 더 **AI에 의존**하고 있다는 사실에서 기인한 두려움일 것입니다.

편안함에 잠식되다보면 도태될 것만 같습니다.

### [1] 마음을 다잡아볼까요

사람마다 마음을 다잡는 방법은 다르겠지만,
저는 스스로 나태해졌거나 안일해졌다고 생각이 들 때면 몹시 고되었던 순간을 떠올리곤 합니다.

생활이 게을러지고 운동이 너무 하기 싫을 때면 군대에 있을 때를,  
일에 대한 열정이 떨어지거나 성장이나 학습에 소홀해지면 창업을 했던 시기를 생각합니다.

최근에는 문득 대학에서 수강했던 '**[CS330] 운영체제 및 실험**' 이라는 강의가 떠올랐습니다.

[pintos](https://casys-kaist.github.io/pintos-kaist/)라는 C언어로 작성된 운영체제 프레임워크를 사용했는데,  
말 그대로 '실행만 되는 수준'의 뼈대만 제공되고 스레드와 스케쥴링, 시스템 콜, 파일 시스템, 가상 메모리 등을 직접 구현하는 프로젝트였습니다.

당시에는 너무 어렵고 힘들었지만 이정도는 해야 개발도 하는거구나 라는 생각을 하곤 했는데,  
돌이켜보니 지금 돌아간다고 해도 못할 것 같습니다. 그정도 고생은 안해도 개발자로 일하는데는 아무 문제도 없더라구요.

```txt
Segmentation fault (core dumped)
```

당시에는 ChatGPT도 없었으니 세그멘테이션 폴트가 발생하면 1~2시간 태우는건 기본이었습니다.  
수정 내역을 찾아다니며 한 줄씩 printf()로 찍어보거나, 쓸데없이 malloc이나 free도 넣어보고, 결국 GDB로 콜스택을 한땀씩 역추적해서 풀어내곤 했습니다.
도무지 해결이 되지 않아 머리를 식히러 나가면 자정이 넘은 야심한 시각에도 전산학과 건물 주변을 방황하며 서성이는 좀비 동지들을 자주 볼 수 있었던 프로젝트였습니다.

모든 순간이 힘들었지만, 아이러니하게도 처음으로 코딩을 하면서 묘한 쾌감을 느낀 경험이었습니다.  
며칠씩 고민해서 **설계**한 내용을 코드로 옮겨보았고 고된 과정이었지만 삐걱거리면서 **돌아가는 무언가**를 만들었던 것이니까요.

### [2] 아무래도 편한게 좋습니다

그에 반해 요새 제가 하고 있는 개발은 매우 '자연어'스럽습니다.  
용어에만 조금 익숙해진다면 기계와 대화하는 느낌이 아니라 '영어'를 읽는 것에 가깝습니다.

아주 편안해보입니다. 저는 아무래도 편한게 좋습니다.

```tsx
import { useState } from "react";

const SimpleButton = () => {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount((prev) => prev + 1)}>like {count}</button>
  );
};
```

프론트엔드 개발을 아예 모르는 사람이 슬쩍 읽어보아도, `count`에 숫자가 저장되고, 버튼을 누르면 previous에서 +1 이 되는 버튼이겠구나 라고 짐작할 수 있습니다.

사실 더 쉽고 편한 방법이 있습니다.  
마치 친구한테 말하듯 AI한테 이렇게 말하면 만들어줍니다.

```txt
클릭하면 +1이 되는 React 컴포넌트를 만들어봐
```

아무튼 원래 이야기로 돌아와서,  
위 코드만 이해하는 데에는 상태 변화로 야기되는 리렌더링, 클릭 이벤트의 버블링/캡처링/위임, 상태 관리 같은 복잡한 개념은 필요하지 않습니다.  
왜냐하면 그 모든 **내부 동작**은 React가 모두 **내부적으로** 알아서 처리해주고 있으니까요.

이러한 추상화는 React라는 라이브러리의 존재 이유이자,  
빠르게 새로운 기능을 개발하고 UI를 만들 수 있는 일종의 기반 시설이라고 생각합니다.  
우리가 물을 마실 때 이 물이 어디서 왔는지 관심 가지지 않고, 수도꼭지의 동작원리를 생각하지 않고, 컵의 제조 공정에 관심이 없는 것처럼요.

토스의 [Frontend Fundamentals](https://frontend-fundamentals.com/code-quality/code/examples/login-start-page.html#%F0%9F%94%8D-%E1%84%83%E1%85%A5-%E1%84%8B%E1%85%A1%E1%86%AF%E1%84%8B%E1%85%A1%E1%84%87%E1%85%A9%E1%84%80%E1%85%B5-%E1%84%8E%E1%85%AE%E1%84%89%E1%85%A1%E1%86%BC%E1%84%92%E1%85%AA)를 읽다가 제 생각을 완벽하게 표현한 문장이 있어 인용해봅니다.

> "왼쪽으로 10걸음 걸어라" 라고 하는 문장이 있어요. 여기에서
>
> “왼쪽”은 “북쪽을 바라보았을 때 90도 돌아간 위치” 를 추상화한 것이고,  
> “90도”는 “한 번의 회전을 360등분한 각의 90배만큼 시초선에 대해 시계 반대 방향으로 돌아간 것” 을 추상화한 것이고,  
> “시계 방향” 의 정의는 “북반구에서 해시계의 바늘이 돌아가는 방향” 을 추상화한 것이에요.

### [3] 그래도 힘들어야 얻는게 있습니다

그럼에도 불구하고, 앞선 예시들과 개발자는 조금 다르다고 생각합니다.

**개발자는 소프트웨어를 설계하고 코드를 작성하여 만드는 사람을 일컫습니다.**

물론 개발자는 React라는 라이브러리를 자유롭게 사용하여 코드를 작성하는 역할도 하지만,  
동시에 필요하다면 React와 같은 라이브러리를 만들 수 있어야 합니다.

단순히 기존 베이스 위에서 비즈니스 레벨이나 서비스의 동작을 설계하는 것을 넘어,  
무에서 유를 창조하고 설계할 수 있어야 하지 않을까요.

나중에는 회사 면접을 볼 때 AI보다 본인이 잘하는 점을 어필해야하는 때가 오지 않을까 싶습니다.  
그 답변이 될 수 있을 내용을 조금씩 미리 준비해보려고 합니다.

AI는 이미 저보다 훨씬 많은 '사용 패턴'을 학습했을 것이고 더 빠르게 많은 코드를 작성할 수 있습니다.
아마 알고리즘과 같은 정해진 문제와 정답이 있는 상황에서는 저보다 몇 배는 빠르게 문제를 풀어낼겁니다.

AI가 더 잘하는 영역이 구현과 속도라면,  
**개발자의 가치는 무엇을 만들 것이고 왜 그렇게 만들어야 하는지를 결정하는 능력**에 있다고 생각합니다.  
답이 이미 정해진 문제를 푸는 것이 아니라, 아직 답이 없는 문제를 정의하고 설계하고 추상화하는 능력입니다.

벌써 힘들고 하기 싫지만 힘들어야 얻는게 있습니다.

26년을 맞이하며 웹 어플리케이션을 구현할 수 있는 '**도구**'를 바닥(scratch)부터 만들어보려 합니다.  
React, Svelte, Vue와 같은 라이브러리, 프레임워크의 구현을 답습하려는 것은 아닙니다.  
설령 결과물이 기존 도구와 같거나 유사한 결과물이 나올지도 모르겠지만,  
그 과정에서의 의사결정과 각 선택의 trade-off를 기록하고 왜 이렇게 만들었는지 기록해보려합니다.

### [4] Do you know Kongnamul?

시작이 반이라고 했으니 오늘은 가볍게 프로젝트의 이름만 정해보고 이만 마치려고 합니다.

저를 힘들게 했던 운영체제인 [pintos](https://casys-kaist.github.io/pintos-kaist/)의 어원을 찾아보니 pintOS가 아닌, 멕시코에서 자주 먹는 핀토콩(pinto)에서 유래했다고 하더라구요.

콩 비슷한게 뭐가 있을까요. 완두콩도 있고 콩으로 만든 두부도 있겠죠.  
찾아보니 **콩나물**은 전세계에서 거의 한국에서만 먹는 식재료라고 하네요.  
그래서 이번 프로젝트의 이름은 `kongnamul`입니다.
